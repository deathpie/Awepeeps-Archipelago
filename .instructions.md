# AI Instructions for Archipelago Workspace

## Maintaining Season READMEs

This workspace contains season folders (e.g., `2026/1. Spring/`) for organizing Archipelago games and configurations. Each season folder should have a `README.md` file that lists the games in a standardized format.

### Process for Creating or Updating a Season README.md

1. **File Location**: Place the `README.md` in the root of the season folder (e.g., `2026/1. Spring/README.md`).

2. **Title**: Use the format `# [Season Name] Games`, e.g., `# Spring 2026 Games`.

3. **Table Structure**:
   - Columns: `Game`, `Info`, `Setup Guide`, `APWorld`, `PopTracker`, `Misc.`
   - Use a markdown table format.

4. **Content Guidelines**:
   - List games in **alphabetical order** by game name.
   - For cells with links, format as `[Game Name Column](link)`. Examples:
     - Info: `[Stardew Valley Info](https://archipelago.gg/games/Stardew%20Valley/info/en)`
     - Setup Guide: `[Stardew Valley Setup Guide](https://archipelago.gg/tutorial/Stardew%20Valley/setup_en)`
     - APWorld: `[Stardew Valley APWorld](https://github.com/example/releases)`
     - PopTracker: `[Stardew Valley PopTracker](https://github.com/example/releases)`
   - Use `-` for empty or unavailable cells.
   - Do not include player names, YAML filenames, timestamps, or other metadata.

5. **Data Sources**:
   - Game info and setup guides: Primarily from `archipelago.gg` (e.g., `https://archipelago.gg/games/[Game]/info/en` and `https://archipelago.gg/tutorial/[Game]/setup_en`).
   - APWorld releases: GitHub releases pages for the respective AP worlds.
   - PopTracker packs: GitHub repositories for PopTracker packs.
   - YAML configs: Stored in the `YAML/` subfolder, but not referenced in the README.

6. **Consistency**: Ensure all season READMEs follow this exact format for easy maintenance and reference.

### Example Table Structure

```
# Spring 2026 Games

| Game | Info | Setup Guide | APWorld | PopTracker | Misc. |
|------|------|-------------|---------|------------|-------|
| Dark Souls III | [Dark Souls III Info](https://archipelago.gg/games/Dark%20Souls%20III/info/en) | [Dark Souls III Setup Guide](https://archipelago.gg/tutorial/Dark%20Souls%20III/setup_en) | - | [Dark Souls III PopTracker](https://github.com/Br00ty/DS3_AP_Maptracker/releases) | - |
| ... | ... | ... | ... | ... | ... |
```

This process ensures that future updates to season READMEs are standardized and efficient.

## Creating New Seasons

When creating a new season folder (e.g., `2026/3. Fall/`), replicate the following structure based on existing seasons:

1. **Main Folder**: Create `2026/3. Fall/` (adjust year and season as needed).

2. **Subfolders**:
   - `APWorld/` (empty, for Archipelago world files)
   - `YAML/` (empty, for player YAML configs)
   - `.secrets/revealed/` (empty, for shareable spoiler logs)
   - `.secrets/hidden/` (empty, for private spoilers; this will be gitignored)

3. **Files** (copy from a previous season and clear contents where applicable):
   - `README.md`: Update title (e.g., "# Fall 2026 Games"), keep table structure but clear rows, set session links to TBD placeholders
   - `host.yaml`: Copy from the most recent previous season (contains host-specific settings, including game and seed settings)

   Note: Do not create `hints.txt`, `room_url.txt`, `game_settings.txt`, or `seed_settings.txt` as this information is now included in the README's Session Links section or consolidated in `host.yaml`.

4. **README Updates**: Include placeholders for Google Sheet and Archipelago Room links, plus navigation links to YAML/, APWorld/, and .secrets/.

5. **YAML Naming**: When adding player configs, name them as `playername_gamename_v0.yaml` for placeholder files, or `playername_gamename_v1.yaml` when a final config is ready. Use hyphens for spaces in the game name portion. The player alias may include spaces or punctuation, but keep the game portion normalized for readability and consistency.

6. **Placeholder YAML Generation**: When generating placeholder player YAMLs from a provided list of players and games, follow these rules:
   - Use the template file from `Templates/` that matches the exact game title if it exists. If it does not exist, create a minimal placeholder YAML based on the requested game name.
   - Use the filename format: `PlayerAlias_Game-Title_v0.yaml`.
     - Example: `AllisonC_Ratchet-and-Clank-Up-Your-Arsenal_v0.yaml` or `Allison C._Ratchet-and-Clank-Up-Your-Arsenal_v0.yaml`.
     - For the game portion, replace spaces with hyphens and keep punctuation as needed for clarity.
   - In the YAML content, set the top-level player name line to `name: PlayerAlias-{player}`.
     - Example: `name: AllisonC-{player}`, `name: Kriegs-{player}`.
   - Set `description` to a clear placeholder description for the game, such as `Placeholder YAML for <Game Title>`.
   - Ensure `game:` matches the exact game title used by the  template or the requested game string.
   - Include a `requires:` section with `version: 0.6.7` for placeholder compatibility.
   - Keep the file name unchanged once created; updates to the YAML should not rename the placeholder file unless the player alias or game title changes significantly.

This ensures the AI can generate placeholder YAMLs consistently from player/game lists while maintaining the correct naming conventions and placeholder behavior.

### Example: Generating Placeholder YAMLs from a Player/Game List

Suppose you provide a list like:
- Kriegs: Dark Souls III
- AllisonC: Ratchet & Clank: Up Your Arsenal
- Max: Hollow Knight

The AI would generate the following placeholder YAML files in the appropriate season's `YAML/` folder:

1. **Kriegs_Dark-Souls-III_v0.yaml**
   - Filename: `Kriegs_Dark-Souls-III_v0.yaml`
   - Content includes: `name: Kriegs-{player}`, `description: Placeholder YAML for Dark Souls III`, `game: Dark Souls III`, etc.
   - Uses the `Templates/Dark Souls III.yaml` as the base if available.

2. **AllisonC_Ratchet-and-Clank-Up-Your-Arsenal_v0.yaml**
   - Filename: `AllisonC_Ratchet-and-Clank-Up-Your-Arsenal_v0.yaml`
   - Content includes: `name: AllisonC-{player}`, `description: Placeholder YAML for Ratchet & Clank: Up Your Arsenal`, `game: Ratchet & Clank: Up Your Arsenal`, etc.
   - Uses the `Templates/Ratchet and Clank 3.yaml` as the base (adapted for the game title), or creates a minimal placeholder if no exact match.

3. **Max_Hollow-Knight_v0.yaml**
   - Filename: `Max_Hollow-Knight_v0.yaml`
   - Content includes: `name: Max-{player}`, `description: Placeholder YAML for Hollow Knight`, `game: Hollow Knight`, etc.
   - Uses the `Templates/Hollow Knight.yaml` as the base.

This example illustrates the filename normalization, player name formatting, and template usage for consistency.

This ensures consistent setup for new seasons, allowing quick scaffolding before populating with specific details.

## Git Operations

The AI assistant must never perform git operations such as `git add`, `git commit`, `git push`, or any other git commands. All git actions should be done manually by humans to ensure proper review and control. The AI can suggest commands or prepare files, but execution of git operations is strictly human-only.
